                          IMPLEMENTATION SUMMARY
                    Unactivated User Login & Activation Flow
═══════════════════════════════════════════════════════════════════════════════

PROBLEM:
────────
Users who didn't activate during email verification were unable to activate 
later when trying to login. The activation page (which is sessionless) couldn't 
get the user's email when they were redirected from the login flow.

SOLUTION:
────────
Store the user's email in browser sessionStorage before redirecting to the 
activation page. The activation page already reads from sessionStorage.

HOW IT WORKS:
─────────────

SCENARIO 1: Email Verification → Immediate Activation
    User clicks email link
    → ConfirmContent verifies token via /api/auth/verify-email
    → ConfirmContent stores email in sessionStorage ✓ (already done)
    → ConfirmContent redirects to /auth/activate
    → ActivateComponent reads email from sessionStorage
    → User activates

SCENARIO 2: Login → Unactivated → Activation
    User tries to login
    → signIn() succeeds, session created
    → checkUserStatusAndRedirect() checks status
    → Detects !isActivationPaid
    → Stores email in sessionStorage ✓ (NEW CODE)
    → Redirects to /auth/activate
    → ActivateComponent reads email from sessionStorage
    → User activates

SCENARIO 3: Return Days Later (Still Unactivated)
    User tries to login again
    → Follows Scenario 2
    → Can complete activation
